## 🧩 题目

在对象赋值时，如果频繁使用「深拷贝」而不是「浅拷贝」，会带来哪些性能和内存方面的影响？在实际项目中，如何平衡拷贝深度与性能？

---

## 💡 解题思路

1. 深拷贝会遍历整个对象图，时间和空间都是 O(n)，大对象代价高
2. 频繁深拷贝会产生大量短生命周期对象，增加 GC 压力
3. 平衡方法：尽量使用不可变数据 + 局部浅拷贝、结构共享、按需深拷贝

---

## ✅ 面试回答（标准版）

1. **深拷贝 vs 浅拷贝的成本差异**  
   - **浅拷贝**（如 `Object.assign({}, obj)`、`{ ...obj }`）：  
     - 只复制第一层属性的引用，嵌套对象仍然指向同一块内存。  
     - 时间复杂度约为 \(O(k)\)，其中 k 为第一层属性数量，内存开销较小。  
   - **深拷贝**：  
     - 需要递归遍历整个对象图，为每个嵌套对象/数组创建新副本。  
     - 时间复杂度和空间复杂度通常为 \(O(n)\)，其中 n 为所有可达节点数量。  
     - 对于大型对象或频繁操作，深拷贝会明显增加 CPU 与内存消耗。  

2. **频繁深拷贝带来的问题**  
   - **性能开销**：  
     - 每次深拷贝都要遍历整个结构，即使只修改了一个很小的字段，也会为全部子对象创建新副本。  
   - **内存与 GC 压力**：  
     - 会产生大量短生命周期对象，占用堆空间，增加垃圾回收频率，可能引发卡顿。  
   - **实现复杂度与 bug 风险**：  
     - 自定义深拷贝实现需要考虑循环引用、特殊类型（Date/RegExp/Map/Set 等），实现不当容易引入隐性 bug。  

3. **如何平衡拷贝深度与性能**  
   - **优先采用「局部浅拷贝 + 结构共享」策略**：  
     - 在不可变数据的模式下，只对需要修改的那一层做浅拷贝，其余层级复用原引用。  

     ```js
     // 只拷贝 user 和 address 层，其余共享
     const newState = {
       ...state,
       user: {
         ...state.user,
         address: {
           ...state.user.address,
           city: 'Shanghai',
         },
       },
     };
     ```

   - **按需深拷贝，而不是默认深拷贝**：  
     - 只有在确实需要完全隔离嵌套对象时（如跨线程、序列化传输、第三方库不可信）才使用深拷贝。  
   - **对超大结构进行拆分**：  
     - 将大的状态拆分为多个相对独立的子模块，避免每次更新都涉及整棵大对象。  
   - **使用专门的不可变数据库或工具**：  
     - 如 Immer，通过「代理 + 差分生成」方式，在开发体验上像是直接修改对象，底层使用结构共享生成新状态，减少手工深拷贝。  

4. **总结**  
   - 深拷贝是一种「简单暴力但代价高」的手段，适合在安全隔离要求较高但数据规模不大的场景使用。  
   - 在大型前端应用中，更推荐采用「不可变数据 + 局部浅拷贝 + 结构共享」的策略，仅在必要时对小范围数据做深拷贝，以平衡性能与安全性。
