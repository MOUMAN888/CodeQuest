## 🧩 题目

如果让你实现一个 Web 端的 AI 助手（类似 ChatGPT），前端整体实现思路是什么？前后端可以采用何种通信方式来实现回答的流式输出？

---

## 💡 解题思路

1. 前端架构：对话列表、输入框、消息状态（发送中/失败）、上下文管理
2. 通信方式：HTTP + SSE、WebSocket，对比优缺点
3. 流式输出实现：后端按 token/句子流式推送，前端逐段追加渲染

---

## ✅ 面试回答（标准版）

1. **AI 助手前端整体设计思路**  
   - UI 结构：  
     - 对话消息列表（气泡 + 角色标识）、输入区域、历史会话列表、重试/停止生成按钮等。  
   - 状态管理：  
     - 消息列表状态（用户消息/助手消息）、消息发送状态（loading、error）、会话上下文（historyId）。  
     - 可使用 React/Vue + 状态管理库（Redux/Zustand/Pinia）或局部 state。  
   - 交互流程：  
     1）用户输入问题 → 点击发送。  
     2）前端追加一条「用户消息」与一条「AI 消息（占位，状态为生成中）」。  
     3）调用后端接口建立流式连接，持续接收 AI 回答片段并更新对应消息。  

2. **实现流式输出的通信方式**  
   - **方案一：HTTP + SSE（Server-Sent Events）**（非常适合 AI 对话）  
     - 前端使用 `EventSource` 或基于 `fetch` 的 ReadableStream，与后端建立 `text/event-stream` 长连接。  
     - 后端按 token 或句子分段推送：  

     ```text
     data: {"delta": "你好"}\n\n
     data: {"delta": "，有什么可以帮你？"}\n\n
     ```

     - 前端在 `onmessage` 回调中不断将 `delta` 追加到当前 AI 消息文本上，实现「逐字/逐句出现」的效果。  
     - 优点：基于 HTTP 容易复用现有网关/鉴权，适合单向流推送。  

   - **方案二：WebSocket**  
     - 建立全双工连接，既可发送消息，也可流式接收回答。  
     - 服务端通过约定的消息格式推送回答分片，前端在 `onmessage` 中累加。  
     - 优点：适合有更多实时交互的场景（如协作、语音/多流）；缺点是基础设施和协议处理更复杂。  

3. **前端流式渲染细节**  
   - 在开始请求时，为 AI 的回复创建一条消息记录：  

   ```js
   const aiMsg = { id, role: 'assistant', content: '', status: 'streaming' };
   ```

   - 每收到一段流式数据：  
     - 将 `delta` 追加到 `aiMsg.content`，触发视图更新。  
     - 自动滚动到列表底部（考虑节流，避免频繁滚动计算）。  
   - 结束标识：  
     - 根据服务端的结束事件（如 `[DONE]`）或连接关闭，将 `status` 置为 `finished`。  
   - 错误与中断处理：  
     - 若网络错误或后端报错，更新消息状态为 `error`，提供「重试」按钮，重新发起同一问题的对话请求。  

4. **总结**  
   - Web 端 AI 助手的关键在于「**良好的对话状态管理 + 流式通信 + 逐步渲染 UI**」。  
   - 流式通信常用 **SSE** 或 **WebSocket**，前者更简单、适合一次请求多段响应的对话场景，后者适合复杂实时交互需求。
