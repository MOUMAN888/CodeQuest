## 🧩 题目

前端项目中为什么要对 Axios 进行封装？拦截器（请求 / 响应）的作用分别是什么？封装 Axios 的核心价值体现在哪些方面？

---

## 💡 解题思路

1. 封装动机：统一配置、统一错误处理、鉴权、重试/取消、类型约束
2. 拦截器：请求拦截器做 token 注入、通用参数，响应拦截器做状态码处理、数据解包
3. 核心价值：降低重复代码、提高一致性与可维护性、方便替换实现

---

## ✅ 面试回答（标准版）

1. **为什么要封装 Axios？**  
   - 直接在各处使用裸 Axios 容易导致：  
     - 重复书写 baseURL、headers、超时等配置。  
     - 错误处理和异常文案风格不统一。  
     - 鉴权逻辑（Token 注入、刷新）散落在各个请求处。  
   - 通过封装一个统一的 HTTP 模块，可以：  
     - 集中管理网络配置与策略。  
     - 提供统一的请求函数（如 `get/post` 封装、业务级泛型类型定义）。  
     - 方便未来替换底层实现（如从 Axios 换到 Fetch）而不影响业务代码。  

2. **请求拦截器的作用**  
   - 在请求发出前集中处理：  
     - **鉴权 Token 注入**：从存储中读取 Token（如 JWT）并写入 `Authorization` 头。  
     - 通用请求头与参数：如语言、版本号、Trace-Id 等。  
     - 请求前的逻辑控制：如在某些情况下直接取消请求或重定向登录页。  
   - 示例：

   ```js
   instance.interceptors.request.use(config => {
     const token = getToken();
     if (token) config.headers.Authorization = `Bearer ${token}`;
     return config;
   });
   ```

3. **响应拦截器的作用**  
   - 在所有响应返回后统一处理：  
     - 统一判断业务状态码（如 `code === 0` 表示成功），不需要每个调用方都手动判断。  
     - 对常见错误码（401 未登录、403 权限不足、500 服务异常）做统一处理（如跳转登录、弹出提示）。  
     - 可以在这里自动刷新 Token 或发起重试逻辑（谨慎使用）。  
     - 对数据进行「解包」，让业务侧直接拿到核心数据结构：  

   ```js
   instance.interceptors.response.use(
     res => res.data, // 或 res.data.data
     error => {
       // 统一错误提示/上报
       return Promise.reject(error);
     }
   );
   ```

4. **封装 Axios 的核心价值**  
   - **降低重复与提升一致性**：  
     - 所有请求共用一套配置和错误处理逻辑，避免每个模块都写一遍。  
   - **提高可维护性与可观测性**：  
     - 统一位置可添加日志、性能监控、埋点等，便于排查问题。  
   - **增强类型与接口约束（在 TS 项目中）**：  
     - 通过泛型与类型定义，约束请求参数与响应数据的结构。  
   - **利于演进**：  
     - 当需要切换请求库或增加新的跨切面逻辑（如重试、熔断）时，只改封装层即可。  

总结：封装 Axios 的本质，是把「网络访问」从零散的 API 调用上升为一层统一的基础设施，在身份鉴权、错误处理、监控和类型安全上提供统一能力，使业务代码更简单、更专注于业务本身。
