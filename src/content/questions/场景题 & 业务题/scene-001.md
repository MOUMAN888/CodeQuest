## 🧩 题目

如果让你从零开发一个「动态路由管理系统」，你会如何设计整体架构？核心功能模块有哪些？前后端的技术选型分别是什么？

---

## 💡 解题思路

1. 明确目标：按用户角色/权限动态生成前端路由和菜单，支持后台配置
2. 架构拆分：路由配置存储与管理、权限模型、前端路由生成、菜单/面包屑、缓存与守卫
3. 技术选型：前端（React/Vue Router、状态管理）、后端（Node/Java/Spring + DB）、接口协议（REST/GraphQL）

---

## ✅ 面试回答（标准版）

1. **系统目标与整体思路**  
   - 目标是实现一个「**可配置的、按权限动态生成的路由和菜单系统**」，让运营/管理员可以在后台配置：  
     - 菜单层级、路由路径、组件映射、权限点（如按钮级权限）等。  
   - 系统需要支持：  
     - 根据不同用户角色返回不同的路由树和菜单树。  
     - 前端基于路由树动态注册路由、渲染菜单、控制页面与按钮可见性。  

2. **核心功能模块设计**  
   - **后端路由/权限配置模块**：  
     - 数据模型：  
       - 路由表：`id`、`parentId`、`path`、`name`、`componentKey`、`meta`（标题、图标、是否缓存、排序等）。  
       - 权限表：角色与路由、按钮权限（如 `canEditUser`）的映射。  
     - 功能：  
       - 管理端 UI：增删改查路由节点、配置角色权限、拖拽调整菜单顺序等。  
       - API：  
         - `GET /api/routes?role=xxx`：返回当前用户可访问的路由树。  
         - `GET /api/permissions`：返回按钮级/资源级权限点。  

   - **前端路由生成模块**（以 Vue/React 为例说明）  
     - 启动时：  
       1）用户登录后，根据 Token 向后端获取当前用户的路由树和权限点。  
       2）通过「组件映射表」将 `componentKey` 映射到实际组件：  
          - 如：`const componentMap = { 'UserList': () => import('@/pages/User/List.vue') }`。  
       3）遍历路由树，动态调用 Vue Router / React Router 的 API 注册路由。  
     - 渲染侧：  
       - 菜单组件根据路由树生成侧边栏/顶部导航。  
       - 利用路由 `meta` 控制面包屑、标签页、多级导航等。  

   - **权限控制模块**  
     - 路由守卫：在全局前置守卫中检查用户是否有访问该路由的权限，没有则重定向到 403/登录页。  
     - 组件级权限：利用指令/HOC/Hook 判断权限点，控制按钮、操作入口的展示/禁用。  

   - **缓存与导航体验**  
     - 路由 `meta` 中可配置是否缓存页面、keep-alive、标签页（多页签）策略。  
     - 利用状态管理（Vuex/Pinia/Redux）记录「已打开路由标签」，支持多标签切换。  

3. **技术选型示例**  
   - **前端**：  
     - 框架：Vue 3 + Vue Router 或 React + React Router。  
     - 状态管理：Pinia/Vuex 或 Redux/Zustand。  
     - UI 组件库：Element Plus、Ant Design 等，快速构建菜单/表单。  
   - **后端**：  
     - 例如 Node.js（NestJS/Express）或 Java（Spring Boot），提供路由/权限管理 API。  
     - 存储：MySQL/PostgreSQL 维护路由/角色/权限关联表。  

4. **总结亮点**  
   - 前后端以「**路由树 + 权限点**」为核心数据模型，前端只负责根据后端下发的数据渲染 UI 和注册路由。  
   - 动态路由与菜单完全可配置，适合中后台多角色、多系统统一管理场景。
