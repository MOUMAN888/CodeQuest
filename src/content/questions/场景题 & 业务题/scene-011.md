## 🧩 题目

前端错误处理常见的方案有哪些？例如全局错误捕获、局部 try/catch、错误边界、监控上报等，它们各自的优缺点是什么？你在项目中一般采用哪种组合方案？

---

## 💡 解题思路

1. 错误类型：运行时 JS 错误、资源加载错误、Promise 未捕获错误、业务错误
2. 处理方式：全局监听（window.onerror / onunhandledrejection / error 事件）、React Error Boundary、局部 try/catch
3. 监控上报：统一封装 SDK，上报到服务端

---

## ✅ 面试回答（标准版）

1. **常见前端错误类型**  
   - 运行时 JS 错误（`ReferenceError`、`TypeError` 等）。  
   - 资源加载错误（图片、脚本、样式加载失败）。  
   - Promise 未捕获异常（UnhandledRejection）。  
   - 业务逻辑错误（接口返回错误码、字段为空等）。  

2. **错误处理的常见方案**  
   - **全局错误捕获**：  
     - `window.onerror`：捕获大部分运行时错误。  
     - `window.addEventListener('error', ...)`：捕获资源加载错误（注意 `event.target` 区分）。  
     - `window.addEventListener('unhandledrejection', ...)`：捕获未处理的 Promise 异常。  
     - 优点：覆盖面广，作为兜底层；缺点：对具体上下文了解有限，只适合作为最后一道防线。  
   - **局部 try/catch**：  
     - 在可能抛错的逻辑（如 JSON 解析、复杂计算、第三方库调用）外层包裹 `try/catch`，可以根据业务做细粒度处理和降级。  
     - 优点：上下文完整，可给用户更友好的反馈；缺点：需要开发者自觉维护，覆盖不全面。  
   - **框架级错误边界（如 React Error Boundary）**：  
     - 在 React 中使用 Error Boundary 捕获子组件渲染中的异常，显示降级 UI 而不是整个页面白屏。  
     - 优点：能避免单个组件错误造成整个应用不可用；缺点：无法捕获事件处理、异步等场景，需要与其他方案配合。  
   - **统一错误监控与上报**：  
     - 封装错误 SDK，在上述捕获点统一上报错误信息（错误类型、堆栈、URL、用户信息、设备信息等）到监控平台（如 Sentry、自建服务）。  
     - 方便在后台聚合分析错误、告警与排查。  

3. **各方案的优缺点与组合实践**  
   - 全局捕获：  
     - ✅ 覆盖面广，能兜底未知错误。  
     - ❌ 对具体业务上下文了解少，只能做简单提示/上报，无法细腻恢复。  
   - 局部 try/catch：  
     - ✅ 精细控制，适合对用户体验要求高的核心路径。  
     - ❌ 容易遗漏，增加代码复杂度。  
   - 错误边界：  
     - ✅ 适合组件级降级，避免整页崩溃。  
     - ❌ 框架相关，无法捕获所有类型错误。  
   - 监控上报：  
     - ✅ 有助于发现线上问题、统计频次、评估影响范围。  
     - ❌ 只能「发现和定位」，不能直接「修复」。  

4. **项目中的推荐组合**  
   - 一般会采用「**局部防御 + 全局兜底 + 统一监控**」的组合：  
     - 核心流程中使用 try/catch 或显式错误处理，提供友好提示与降级。  
     - 使用 Error Boundary 等防止单点组件错误导致整页白屏。  
     - 在全局注册 onerror / unhandledrejection 监听，将所有未捕获错误上报监控平台进行聚合和告警。  

总结：前端错误处理不只是捕获和打印，更重要的是通过组合使用多种方案，在保证用户体验的前提下做到「可发现、可定位、可量化」，并形成完善的线上监控体系。
