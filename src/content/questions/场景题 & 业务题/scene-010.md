## 🧩 题目

如果让你设计一个类似 Figma 的多人在线画图工具的前端架构，你会如何设计？如何处理实时协作、操作同步以及冲突解决问题？

---

## 💡 解题思路

1. 前端架构：画布渲染（Canvas/SVG/WebGL）、状态管理、工具与层系统
2. 实时协作：WebSocket/SSE 通信、操作日志（OP）同步、房间与光标同步
3. 冲突解决：基于 CRDT 或 OT 的最终一致性策略

---

## ✅ 面试回答（标准版）

1. **前端整体架构**  
   - 渲染层：  
     - 使用 Canvas 2D / WebGL / SVG（视需求而定）渲染形状、连线、文本等。  
     - 支持图层、选中框、缩放/平移、对齐辅助线等。  
   - 状态管理：  
     - 全局使用状态管理库（如 Redux/Zustand/MobX）管理文档结构（节点树）、选中状态、历史栈。  
     - 本地维护「操作队列」，用于撤销/重做与与服务器同步。  
   - 工具与插件系统：  
     - 将画笔、矩形、文本等抽象为「工具」，统一接口方便拓展。  

2. **实时协作与操作同步**  
   - 通信方式：  
     - 使用 WebSocket 建立与协作服务器的长连接，每个文档对应一个「房间」。  
   - 同步粒度：  
     - 不直接同步整个文档快照，而是同步「增量操作」（Operation），如「在节点 X 上修改位置为 (x,y)」。  
     - 每个操作携带：操作类型、目标对象、变更内容、客户端 id、时间戳/版本号。  
   - 前端流程：  
     1）用户本地操作 → 生成操作对象并立即在本地应用（乐观更新）。  
     2）将操作通过 WebSocket 发送到服务器。  
     3）服务器将操作转发给其他在线客户端（经过 OT/CRDT 处理后），各客户端按顺序应用操作更新视图。  
   - 光标与选区同步：  
     - 定期同步当前用户的光标位置、选中对象 id、昵称/颜色，用于显示「协作者光标」。  

3. **冲突解决与一致性（OT / CRDT 简要说明）**  
   - 多人同时编辑同一对象时会产生写冲突，为保证文档最终一致，需要冲突解决算法：  
   - **OT（Operational Transformation）**：  
     - 记录操作序列，根据操作之间的依赖与顺序，对后来的操作进行「变换」，使其在不同客户端上的效果一致。  
     - 常用于文本编辑器与协作白板。  
   - **CRDT（Conflict-free Replicated Data Type）**：  
     - 将文档建模为一种特殊的数据结构，每个操作都是可交换的，通过合并副本自动收敛到同一状态。  
     - 对实现和理解要求更高，但在离线/弱网场景下优势明显。  
   - 在简化场景下，也可以采用「服务器权威状态 + 乐观更新 + 回滚」的方式：  
     - 客户端先本地应用并上报操作，服务器按顺序应用并下发最终状态，如有冲突则让客户端回滚并重放。  

4. **其他需要考虑的点**  
   - undo/redo：  
     - 本地维护操作栈，协作时需要与远端操作序列协同，通常和 OT/CRDT 紧密相关。  
   - 性能优化：  
     - 大量图形对象时使用虚拟化或分层渲染，减少重绘；合理利用 GPU。  
   - 离线与断网：  
     - 可在本地缓存操作，恢复连接后再同步。  

总结：多人在线画图工具的前端核心在于「高性能画布渲染 + 可重放的操作模型 + 实时通信 + 正确的冲突解决策略」，常见实现会基于 WebSocket + OT/CRDT 的组合来实现多人协作的一致性。
