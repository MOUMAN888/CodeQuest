## 🧩 题目

小程序开发与传统 Web 开发的核心区别是什么？在小程序开发中需要特别注意哪些兼容性或平台差异问题？

---

## 💡 解题思路

1. 运行环境不同：小程序容器 vs 浏览器，双线程（逻辑层/视图层）、setData
2. 技术栈差异：WXML/WXSS/组件体系、API 封装、本地能力
3. 兼容性点：不同平台（微信/支付宝/抖音）、不同基础库版本差异

---

## ✅ 面试回答（标准版）

1. **运行环境与架构差异**  
   - **Web**：运行在浏览器中，JS 直接操作 DOM，使用标准 HTML/CSS/JS 与浏览器 API。  
   - **小程序**：运行在各大平台提供的容器中（微信、支付宝、字节等），通常采用「**逻辑层 + 视图层分离**」的架构：  
     - 逻辑层运行在独立的 JSCore 环境，不直接接触真实 DOM。  
     - 视图层使用 WXML/自定义组件，由 WebView 或原生渲染引擎负责。  
     - 状态更新需要通过 `setData` 在两层之间同步，频繁/大体积 `setData` 会造成性能问题。  

2. **技术栈与能力差异**  
   - 视图与样式：  
     - 使用 WXML/WXSS（或平台等价语言）描述结构与样式，而不是标准 HTML/CSS。  
     - 提供一套平台内置组件（`view`、`button`、`scroll-view` 等），部分组件为原生实现（性能更优）。  
   - API 与系统能力：  
     - 小程序提供封装好的 API（`wx.request`、`wx.login`、`wx.getUserInfo`、文件/相机/位置等），与浏览器的 `fetch`、`localStorage`、`navigator` 等不同。  
   - 打包与发布：  
     - 小程序有自己的构建与上传流程，包体积与分包结构有严格限制。  

3. **兼容性与平台差异需要注意的点**  
   - **不同平台的差异**：  
     - 微信/支付宝/抖音/百度小程序在组件、API、生命周期等方面都存在差异。  
     - 若要多端复用，需要使用跨端框架（Taro、uni-app 等）或封装一层适配。  
   - **基础库版本兼容**：  
     - 小程序的基础库版本决定可用 API 与行为，老设备上基础库可能落后，需要考虑向下兼容或在运行时做能力检测。  
   - **渲染与性能限制**：  
     - DOM 节点数、WXML 层级、`setData` 调用频率都有性能红线；复杂动画和大数据渲染需谨慎设计。  
   - **网络与安全限制**：  
     - 请求域名需在小程序管理后台配置白名单。  

4. **总结**  
   - 小程序本质上是运行在特定容器中的「微应用」，虽然与 Web 类似，但在运行环境、组件体系、API 和发布模式上有明显差异。  
   - 开发时需特别关注：`setData` 使用、平台能力与差异、基础库兼容，以及性能与包体积限制。
