## 🧩 题目

如果让你重构一个老旧的前端模块，你会遵循哪些步骤？如何在重构过程中保证功能的安全性和整体项目的稳定性？

---

## 💡 解题思路

1. 步骤：调研与梳理现状 → 明确目标与边界 → 衍生用例与测试 → 分阶段重构 → 验证与回滚策略
2. 安全性：自动化测试、防护开关、灰度发布、充分 code review

---

## ✅ 面试回答（标准版）

1. **前期调研与目标确认**  
   - 理解模块当前的业务职责与使用场景：谁在调用、关键流程是什么、有哪些隐含依赖。  
   - 评估痛点：如代码难维护、性能瓶颈、技术栈过旧（jQuery → Vue/React）等。  
   - 明确重构目标与边界：  
     - 是「等价重写」还是「顺带优化交互」？  
     - 是否只影响某个子页面或整个系统？  

2. **梳理接口与编写测试/用例**  
   - 梳理当前模块对外暴露的 API/props/事件、对后端接口的依赖。  
   - 尽量在重构前补上一层自动化测试：  
     - 单元测试（核心逻辑、数据处理函数）。  
     - 简单的集成/E2E 测试（关键路径：创建/编辑/提交等）。  
   - 若暂时无法上自动化测试，至少整理「手动回归用例清单」。  

3. **设计新实现与分阶段重构**  
   - 在新技术栈或新架构中设计等价模块：组件划分、状态管理、接口适配方式等。  
   - 尽量 **分阶段重构**，避免「一次性全部推倒重来」：  
     - 先抽离公共逻辑为独立模块（如数据处理、校验），减少与旧视图层的耦合。  
     - 再将视图层逐步迁移到新实现，必要时使用「适配层」同时兼容新旧调用方式。  

4. **保证安全性与稳定性的具体措施**  
   - **兼容层/适配层**：  
     - 为旧调用方保留相同的接口签名和行为，内部逐步切换到新实现。  
   - **灰度发布与开关控制**：  
     - 使用 feature flag：部分用户或环境（如内网/灰度环境）先使用新模块，观察日志和反馈。  
     - 出现严重问题可一键切换回旧实现。  
   - **监控与日志**：  
     - 针对重构模块增加错误监控、性能指标上报，方便发现异常。  
   - **充分的 Code Review 与联调**：  
     - 邀请熟悉业务的同事参与 Review，重点关注边界情况与兼容性。  

5. **上线后的验证与持续改进**  
   - 上线后严格关注监控面板和用户反馈，及时修复问题。  
   - 在确认稳定后，逐步清理旧代码和临时兼容逻辑。  

总结：重构的关键不是「把代码写漂亮」本身，而是在**明确目标、保持行为等价、建立测试与回滚策略**的前提下，逐步改善老代码的可维护性和性能，确保业务风险可控。
