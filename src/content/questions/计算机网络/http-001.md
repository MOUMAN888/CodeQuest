## 🧩 题目

HTTP 中 `GET` 和 `POST` 的核心区别是什么？在语义、缓存、中间层处理、安全性和使用限制等方面分别有哪些特点？各自适用于什么场景？

---

## 💡 解题思路

1. 从语义出发：GET 是安全、幂等的获取资源；POST 是非幂等的提交/修改资源
2. 协议与实现：报文结构其实都可以带 body，差异更多来自约定与中间件处理
3. 浏览器与缓存代理的行为差异：GET 更容易被缓存、预取和重试
4. 安全性与限制：URL 长度限制、敏感数据暴露、CSRF 风险

---

## ✅ 面试回答（标准版）

1. **语义上的核心区别**  
   - `GET`：表示**获取资源**，是安全（不改变服务器状态）、幂等（多次调用效果相同）的操作；适用于查询、读取数据的场景。  
   - `POST`：表示**提交数据以创建或修改资源**，不保证幂等，多次调用可能产生多条记录或多次副作用；适用于表单提交、创建/更新数据、执行动作等。  
   语义决定了中间件（代理、网关、浏览器）可以对 GET 做更多缓存、重试等优化，而对 POST 会更谨慎。  

2. **从协议角度看请求报文**  
   - HTTP 协议本身并不强制「GET 不能有 body，POST 必须有 body」，只是：  
     - 实际实现中，**GET 请求几乎总把参数放在 URL 查询串中**，body 通常被忽略  
     - **POST 通常带有请求体**（form、JSON 等），用于传递表单数据、JSON Payload 等  
   - 因此「GET 无 body，POST 有 body」更接近事实使用习惯，而不是严格协议限制。  

3. **缓存与中间层处理差异**  
   - GET 请求天然更容易被：  
     - 浏览器、CDN、代理缓存（结合 `Cache-Control`、`ETag` 等）  
     - 透明地重试或预取（如浏览器预加载）  
   - POST 请求默认**不会被缓存**（除非明确指定），也不易被无风险重试，因为可能产生副作用。  
   - 因此：  
     - **查询接口**应优先设计为 GET，便于缓存和调试  
     - **写操作**应使用 POST/PUT/PATCH/DELETE 等语义更明确的方法  

4. **安全性与使用限制**  
   - **URL 长度限制**：  
     - GET 的参数放在 URL 中，受浏览器和服务器对 URL 长度的限制（常见在 2KB~8KB 量级），不适合传输大量数据  
     - POST 数据在 body 中，没有严格大小上限（但会受服务器配置限制）  
   - **信息暴露**：  
     - GET 参数出现在地址栏、浏览器历史、访问日志中，不适合放敏感信息（如密码、Token 等）  
     - POST body 相对不那么显眼，但依然要通过 HTTPS 保护传输安全  
   - CSRF 风险：  
     - 理论上任何会改变服务器状态的请求都应该防 CSRF  
     - 实践中通常将「有副作用的操作」限制为 POST/其他写方法，并配合 CSRF Token、防重复提交等机制  

5. **典型使用场景**  
   - 使用 **GET**：  
     - 搜索、筛选、分页查询等「读取」操作（例如 `/api/users?page=2&role=admin`）  
     - 适合作为带参数的可分享链接  
   - 使用 **POST**：  
     - 提交表单、新建资源：注册、登录、创建订单等  
     - 复杂查询条件、批量操作请求体较大时（可设计为 POST 请求的「查询接口」）  

总结来说：GET 与 POST 的**关键区别在于语义和中间件处理方式**，而不是简单的「一个在 URL，一个在 body」。设计 API 时应遵循「读用 GET、写用 POST/PUT/PATCH/DELETE」的 REST 风格，同时考虑缓存、安全和调试体验。
