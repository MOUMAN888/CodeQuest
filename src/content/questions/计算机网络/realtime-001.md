## 🧩 题目

SSE（Server-Sent Events）与 WebSocket 有什么区别？为什么在 AI 对话等场景中常常采用 SSE？WebSocket 的性能开销大概如何？

---

## 💡 解题思路

1. 先对比协议特性：单向 vs 双向、基于 HTTP vs 独立协议
2. SSE 的特点：基于 HTTP 长连接、服务器单向推送、易于穿透代理、浏览器原生支持
3. WebSocket 的特点：全双工通信、协议升级、适合频繁双向交互
4. 结合 AI 对话：请求-响应流式输出，多数是「客户端发一次请求，服务端持续推送结果」 → SSE 更简单够用
5. 性能角度：WebSocket 连接建立成本略高，但长连接下单条消息开销小；两者差异更多在使用场景与基础设施兼容性

---

## ✅ 面试回答（标准版）

1. **SSE 与 WebSocket 的基本区别**  
   - **SSE（Server-Sent Events）**：  
     - 基于 **HTTP 协议之上的单向服务器推送**：客户端通过普通 HTTP 请求建立长连接，服务端以 `text/event-stream` 格式持续向客户端发送事件流。  
     - 从浏览器角度看，是「服务端 → 客户端」的单向流，客户端不能通过这条通道反向发送消息（仍需普通 HTTP 请求）。  
   - **WebSocket**：  
     - 通过 HTTP 握手后升级为 `ws://` 或 `wss://` 协议，建立一个**全双工的持久连接**。  
     - 客户端和服务端都可以在任意时刻主动发送消息，适合频繁的双向实时通信（聊天室、协作编辑、游戏等）。  

2. **SSE 的特点与优劣**  
   - 优点：  
     - 基于标准 HTTP/1.1 协议，使用简单（浏览器里用 `EventSource` 即可）、容易穿透代理/防火墙及配合现有基础设施。  
     - 原生支持自动重连、事件 ID 等特性。  
     - 对于「服务器单向推送」的场景非常合适，如进度更新、通知、日志流等。  
   - 缺点：  
     - 只支持单向推送，客户端要发消息仍需单独的 HTTP 请求。  
     - 旧浏览器兼容性略差，在移动端网络不稳定场景下需要注意断线重连策略。  

3. **WebSocket 的特点与性能开销**  
   - 优点：  
     - 全双工通信，适合高频、低延迟的实时交互场景（IM、在线游戏、协同编辑等）。  
     - 建立连接后，每条消息头部开销较小，相比频繁发 HTTP 请求更轻量。  
   - 开销与复杂度：  
     - 连接建立阶段需要额外的握手和协议升级，连接维持需要心跳和资源管理，对服务器和网关有一定压力。  
     - 对负载均衡、反向代理（如 Nginx）、鉴权与扩缩容的要求相对更高，需要专门的基础设施支持。  

4. **为什么 AI 对话场景常用 SSE？**  
   - 典型 AI 对话交互模式是：  
     - 客户端发起一次请求（问题 + 上下文），  
     - 服务器以「流式」方式逐步推送生成的回答文本（token 流），  
     - 结束后关闭连接或等待下次请求。  
   - 这类场景中：  
     - 通信模式更接近「**一次请求，多段响应的单向流**」，双向实时通信需求不强。  
     - 基于 HTTP 的 SSE 更容易复用现有 API 网关、鉴权、日志、监控等系统，不需要为 WebSocket 额外配置基础设施。  
     - SSE 数据以文本流形式发送，前端处理简单（按行解析、渐进渲染），非常贴合流式输出的需求。  
   - 因此，**SSE 在 AI 对话场景下往往已经足够，且实现成本和运维复杂度更低**。  

5. **如何选择？**  
   - 需要高频双向实时通信（游戏、聊天、协作编辑） → 更适合 WebSocket。  
   - 以「服务器单向推送」为主，如日志流、进度条、AI 流式输出 → 优先考虑 SSE。  
   - 在基础设施对 WebSocket 支持不佳、希望尽量复用现有 HTTP 栈时，SSE 往往是更务实的选择。
