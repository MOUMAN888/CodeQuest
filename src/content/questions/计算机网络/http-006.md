## 🧩 题目

HTTP/1.1 和 HTTP/2 的核心区别是什么？HTTP/2 在性能上主要做了哪些优化？

---

## 💡 解题思路

1. HTTP/1.1 的特点与瓶颈：队头阻塞、多连接、文本协议
2. HTTP/2 核心特性：二进制分帧、多路复用、首部压缩、服务器推送、优先级
3. 概述对前端实践的影响：域名分片不再必要、合并资源策略变化

---

## ✅ 面试回答（标准版）

1. **HTTP/1.1 的特点与瓶颈**  
   - 默认支持持久连接（`Connection: keep-alive`），一个 TCP 连接可以复用来发送多个请求，但**同一连接上仍是串行的请求-响应**。  
   - 为了提升并发，浏览器通常会为同一域名建立多个 TCP 连接（例如 6 个），这会带来 TCP 握手、慢启动、队头阻塞等开销。  
   - 报文是基于文本的，首部重复且体积不小。  

2. **HTTP/2 的核心特性**  
   - **二进制分帧（Binary Framing）**：  
     - HTTP/2 将请求和响应拆分成更小的二进制帧，通过流 ID 区分不同的请求/响应流。  
   - **多路复用（Multiplexing）**：  
     - 在一个 TCP 连接上并行发送多个请求和响应，帧可以交错传输，解决了 HTTP/1.1 的队头阻塞问题（应用层）。  
   - **首部压缩（HPACK）**：  
     - 使用静态/动态字典对 HTTP 头部进行压缩，减少重复的头部数据（如 Cookie、User-Agent 等），降低带宽开销。  
   - **服务器推送（Server Push）**：  
     - 服务器可在客户端请求某资源时，主动推送它可能接下来需要的资源（如 CSS/JS），减少请求往返次数（但实践中很多框架已不再推荐滥用）。  
   - **流优先级与依赖**：  
     - 可以为不同请求设置优先级，使重要资源优先传输。  

3. **性能优化点总结**  
   - **单连接多路复用**：  
     - 减少 TCP 连接数量，降低握手和慢启动开销。  
     - 避免「一个请求慢导致后面请求都堵在后面」的队头阻塞（应用层，底层 TCP 仍可能有队头阻塞问题）。  
   - **头部压缩**：  
     - 对于大量短小请求，首部压缩能显著降低冗余数据传输。  
   - **服务器推送**（谨慎使用）：  
     - 理论上可减少关键资源的往返延迟，但需要与缓存和优先级策略配合良好，否则可能适得其反。  

4. **对前端工程实践的影响**  
   - 在 HTTP/1.1 时代，常见的优化手段包括：  
     - 域名分片（Sharding）以突破浏览器单域名连接数限制  
     - 将小文件合并为大文件（CSS Sprite、JS/CSS 合并）以减少请求数  
   - 在 HTTP/2 下：  
     - 多路复用减少了「请求数」本身的性能压力，过度合并可能反而影响缓存粒度与更新策略。  
     - 域名分片可能会降低效果（打散了多路复用能力）。  

一句话：HTTP/2 通过**二进制分帧 + 多路复用 + 头部压缩 +（可选）服务器推送**显著改善了 HTTP/1.1 在连接复用和带宽利用上的瓶颈，让一个 TCP 连接就能高效承载大量请求。
