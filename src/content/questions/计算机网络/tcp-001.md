## 🧩 题目

TCP 的三次握手和四次挥手分别是怎样的具体步骤？为什么建立连接需要三次握手，而关闭连接通常是四次挥手？

---

## 💡 解题思路

1. 画出时间线：客户端（C）与服务端（S）之间的报文交换
2. 三次握手：SYN → SYN+ACK → ACK，确认双方发送/接收能力正常
3. 四次挥手：FIN/ACK 分开发送，半关闭状态
4. 解释为什么不是两次握手、为什么关闭时需要四次以及 TIME_WAIT 的作用

---

## ✅ 面试回答（标准版）

1. **三次握手（建立连接）**  
   假设客户端为 C，服务端为 S：  
   1）**第一次握手**：C → S，发送 `SYN` 报文（SYN=1，seq=x），表示「我想发起连接，请求同步序列号」。  
   2）**第二次握手**：S → C，发送 `SYN + ACK` 报文（SYN=1，ACK=1，ack=x+1，seq=y），表示「我收到了你的请求，也准备好了」。  
   3）**第三次握手**：C → S，发送 `ACK` 报文（ACK=1，ack=y+1），表示「我也收到了你的确认」，连接正式建立。  

   **为什么需要三次？**  
   - 核心是保证**双方的「发送能力」和「接收能力」都是正常的**：  
     - 第一次握手：C → S，C 的发送、S 的接收没问题。  
     - 第二次握手：S → C，S 的发送、C 的接收没问题。  
     - 第三次握手：C → S，确认 C 收到了 S 的确认，若缺少这一步，S 无法确认 C 是否已准备好，容易出现「失效连接请求」导致资源浪费等问题。  

2. **四次挥手（关闭连接）**  
   TCP 链接是**全双工**的，关闭时两端的发送方向需要分别关闭。仍以 C 主动关闭为例：  
   1）**第一次挥手**：C → S，发送 `FIN` 报文，表示「我这边不再发送数据了，但还能接收你的数据」。  
   2）**第二次挥手**：S → C，发送 `ACK` 报文，确认收到 FIN，此时 S 进入半关闭状态（从 C 到 S 的方向关闭）。  
   3）**第三次挥手**：当 S 也不再需要发送数据时，S → C 发送 `FIN` 报文，表示「我也发完了」。  
   4）**第四次挥手**：C → S，发送 `ACK` 报文，确认收到 S 的 FIN。此后 C 进入 `TIME_WAIT` 状态一段时间，再真正关闭。  

   **为什么是四次？**  
   - 因为关闭「发送方向」和「接收方向」是相互独立的：  
     - C 关闭自己的发送方向（FIN），S 只是确认这一动作（ACK），此时 S 仍可能有数据要发给 C；  
     - 等 S 也发完数据，才发送自己的 FIN，再由 C 最后 ACK；  
   - 因此两个方向各需要一个 FIN + 一个 ACK，总共四步。  

3. **TIME_WAIT 的作用（顺带说明）**  
   - 主动关闭方在发送最后一个 ACK 后会进入 `TIME_WAIT` 状态，等待约 2 * MSL 的时间。  
   - 作用：  
     - 确保对方收到最终 ACK（若对方未收到会重发 FIN，这时还能重发 ACK）。  
     - 避免旧连接中的延迟报文影响后续使用同一四元组的新连接。  

总结：三次握手用于在建立连接前「彼此确认对方的收发能力」并避免历史报文干扰；四次挥手则是由于 TCP 的全双工特性，需要对两个方向的关闭分别进行确认，因此需要四步。
