## 🧩 题目

JWT（JSON Web Token）的实现原理是什么？Token 是如何生成与验证的？在实际应用中有哪些常见的安全风险与防护要点？

---

## 💡 解题思路

1. JWT 结构：Header.Payload.Signature（Base64URL 编码）
2. 生成流程：编码头/载荷 + 使用密钥和算法（如 HMAC-SHA256）计算签名
3. 验证流程：解析前两段、重算签名比对、校验过期等声明
4. 安全风险：密钥泄露、算法降级攻击、过期/注销问题、存储位置（XSS）、重放攻击

---

## ✅ 面试回答（标准版）

1. **JWT 的基本结构**  
   一个典型的 JWT 由三部分组成，通过 `.` 连接：  

   ```text
   Header.Payload.Signature
   ```

   - **Header**：头部信息（JSON），如签名算法、类型：`{ "alg": "HS256", "typ": "JWT" }`  
   - **Payload**：负载（JSON），存放用户信息和一些标准声明（如 `sub`、`exp`、`iat`、`iss` 等）  
   - **Signature**：签名，用于防止 Token 被篡改  
   这三部分分别进行 Base64URL 编码后用 `.` 拼接得到最终的 Token。  

2. **JWT 的生成流程**  
   1）服务端根据业务数据构造 Header 和 Payload：  
      - 设置签名算法（如 HS256）、过期时间 `exp`、用户 ID 等。  
   2）对 Header 和 Payload 分别进行 Base64URL 编码：`base64url(header)`、`base64url(payload)`。  
   3）按 `"header.payload"` 拼接，中间用 `.` 分隔。  
   4）使用事先约定的密钥 key 和算法对这段字符串进行签名，例如 HMAC-SHA256：  

   ```text
   signature = HMACSHA256(base64url(header) + "." + base64url(payload), key)
   ```

   5）对签名结果进行 Base64URL 编码，得到第三段。  
   6）最终 Token 即：`base64url(header).base64url(payload).base64url(signature)`。  

3. **JWT 的验证流程**  
   1）接收到 Token 后，服务端首先按 `.` 分割为三部分。  
   2）Base64URL 解码 Header 和 Payload，检查声明（如 `iss`、`aud`、`exp` 是否过期）。  
   3）使用同样的算法和密钥对 `"header.payload"` 重新计算签名，并与 Token 中的 Signature 做对比：  
      - 相等 → 说明 Token 未被篡改且来源可信（前提是密钥未泄露）。  
      - 不相等 → Token 无效。  
   4）若签名和声明都校验通过，即可信任 Payload 中的身份信息并进行后续授权。  

4. **常见安全风险与防护要点**  
   - **密钥泄露**：  
     - 若对称密钥（HS256）泄露，攻击者可以伪造任意合法 JWT。  
     - 需要妥善保管签名密钥，建议使用非对称算法（RS256）并将私钥只保存在认证服务。  
   - **算法降级攻击**：  
     - 历史上曾有服务端错误地允许 `alg: "none"` 或算法切换漏洞，导致签名被绕过。  
     - 应在服务端**强制指定允许的签名算法**，忽略 Token 中的 `alg` 声明。  
   - **过期与注销问题**：  
     - JWT 通常是无状态的，一旦签发，直到过期前服务端无法单独撤销某个 Token。  
     - 解决方式：缩短过期时间、引入刷新 Token（Refresh Token）、维护黑名单或 Token 版本号（在数据库中存放当前有效版本）。  
   - **存储位置与 XSS 风险**：  
     - 将 JWT 储存在 `localStorage` 容易被 XSS 窃取；  
     - 更安全的方式通常是使用 **HttpOnly + Secure Cookie** 存 Token 或 Access Token。  
   - **重放攻击**：  
     - 攻击者窃取合法 Token 后，在有效期内重复使用。  
     - 可结合设备指纹、IP 限制、一次性 Token、重要操作二次验证等方式减轻风险。  

总结：JWT 是通过「Header + Payload + 签名」的三段式结构实现的无状态认证凭证，生成时用密钥对内容签名，验证时重算签名并检查声明。但要真正安全地使用 JWT，需要严格管理密钥、控制有效期、妥善存储 Token 并设计好注销与刷新策略。
