## 🧩 题目

`Object.assign` 和对象扩展运算符（`...`）有什么区别？它们是深拷贝还是浅拷贝？如果要实现真正的深拷贝，有哪些常见方案？

---

## 💡 解题思路

1. `Object.assign(target, ...sources)` 与 `{ ...obj }` 的行为本质上都是**浅拷贝**
2. 指出它们只会拷贝第一层属性引用，对嵌套对象/数组仍然共享引用
3. 对比两者在原型、访问器属性、枚举性上的一些细节差异（简要）
4. 列出深拷贝的几种实现方式及优缺点：`JSON`、手写递归、`structuredClone`、第三方库

---

## ✅ 面试回答（标准版）

1. **`Object.assign` 与扩展运算符的本质**  
   - `Object.assign(target, ...sources)` 会把源对象中**可枚举的自有属性**拷贝到目标对象上，返回目标对象  
   - 对象扩展运算符：`{ ...obj }` 在大多数实现中本质也是对源对象的可枚举自有属性进行**浅拷贝**  
   - 两者效果在简单场景下非常接近，如：

   ```js
   const obj = { a: 1, b: { c: 2 } };
   const copy1 = Object.assign({}, obj);
   const copy2 = { ...obj };
   ```

   - `copy1` 和 `copy2` 中的 `b` 属性都仍然指向**同一个嵌套对象**  

2. **浅拷贝 vs 深拷贝**  
   - **浅拷贝**：只复制对象的第一层属性，如果属性值是引用类型，仅复制其引用（指针）  
   - **深拷贝**：递归复制所有层级的属性，为嵌套对象/数组创建全新的引用  
   - `Object.assign` 和 `{ ...obj }` 都是**浅拷贝**：

   ```js
   const obj = { a: { n: 1 } };
   const copy = { ...obj };
   copy.a.n = 2;
   console.log(obj.a.n); // 2，被一起改了
   ```

3. **两者的一些细微差别（了解即可）**  
   - 在处理访问器属性（getter/setter）时：
     - `Object.assign` 会将源对象的访问器属性**转为目标对象的普通数据属性**（即调用 getter 获取当前值再赋值）  
     - 展开运算符在规范中要求更严格地遵循「拷贝属性描述符」的规则，一些场景下行为略有不同  
   - 原型链：
     - `{ ...obj }` 创建的是一个**全新的普通对象**，原型为 `Object.prototype`  
     - `Object.assign` 可以把属性拷贝到任何目标对象（包括已经带有自定义原型的对象）  

4. **常见的深拷贝实现方式**  
   - **方式一：`JSON.parse(JSON.stringify(obj))`**  
     - 优点：简单粗暴，一行搞定  
     - 缺点：  
       - 会丢失 `function`、`Symbol`、`undefined` 等类型  
       - `Date` 会变成字符串，`RegExp` 会变成空对象  
       - 会抛弃原型链信息  
   - **方式二：手写递归深拷贝**  
     - 遍历对象和数组，遇到引用类型就递归复制，注意：  
       - 处理数组/对象的区分  
       - 处理循环引用（如使用 Map 记录已拷贝过的对象）  
   - **方式三：使用原生 `structuredClone`（现代环境）**  

     ```js
     const copy = structuredClone(obj);
     ```

     - 支持大部分内置类型（`Date`、`Map`、`Set`、`ArrayBuffer` 等），能正确处理循环引用  
   - **方式四：使用成熟库**  
     - 如 Lodash 的 `_.cloneDeep`，在复杂场景下更可靠  

总结：  
**`Object.assign` 和对象展开运算符都是浅拷贝，只适合一层结构的拷贝**。遇到嵌套对象或需要完全隔离引用的场景，应使用 `structuredClone`、手写递归或第三方库实现真正的深拷贝。
