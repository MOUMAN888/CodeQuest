## 🧩 题目

`[] == []` 的结果是什么？为什么？请结合 JavaScript 的隐式类型转换和相等性比较规则进行分析。

---

## 💡 解题思路

1. 回答结果：`[] == [] // false`
2. 说明「引用比较」：引用类型用 ==
   时比较的是引用地址
3. 引出对象到原始值的转换规则（`ToPrimitive` 和 `valueOf` / `toString`），顺带解释如 `[] == ![]`
4. 总结：基本类型按值比较，引用类型按地址比较，`==` 还有额外的类型转换

---

## ✅ 面试回答（标准版）

1. **先给出结论**  

```js
[] == [] // false
```

2. **根本原因：引用类型按引用地址比较**  
   - 在 JavaScript 中，数组是对象的一种，属于**引用类型**  
   - 不管是 `==` 还是 `===`，对对象来说，比较的是它们在堆中的**引用（地址）**是否相同，而不是结构是否相同  
   - 表达式 `[] == []` 中的两个数组字面量是**两个不同的实例**，它们在内存中有不同的地址，因此比较结果为 `false`  

3. **顺带理解对象与原始值的隐式转换**  
   - 在某些场景下，`==` 会触发对象到原始值的转换，但 **对象与对象比较时不会转换为原始值，而是直接比较引用**  
   - 比如一个经典例子：

   ```js
   [] == ![]; // true
   ```

   分析：
   - `![]`：数组对象是 truthy，取反后为 `false`  
   - 表达式变为：`[] == false`  
   - `==` 规则：一边是对象，一边是布尔值 → 先把布尔值转换为数字：`false -> 0`  
   - 再比较：`[] == 0` → 对象与数字比较，需要将对象转为原始值：
     - `[].toString()` 得到 `''`，再转数字得到 `0`  
   - 最终比较 `0 == 0`，结果为 `true`  

   这个例子说明，`==` 规则非常复杂，容易出坑。  

4. **总结与实践建议**  
   - 对于对象（包括数组、函数等），**比较的是引用地址**，而不是内容本身  
   - `[] == []`、`{} == {}`、`new Date() == new Date()` 都是 `false`  
   - 若要比较数组/对象内容是否相等，需要自己实现「深度比较」逻辑或使用第三方库（如 Lodash 的 `_.isEqual`）  
   - 在工程实践里，**尽量使用 `===` 而不是 `==`**，除非对 `==` 的隐式转换规则非常熟悉并、且有明确意图  

一句话：`[] == []` 为 `false`，因为它们是两个不同的引用，哪怕内容一样，引用类型也不会因此被判定为相等。
