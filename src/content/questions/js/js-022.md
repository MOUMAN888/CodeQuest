## 🧩 题目

在调用多个模型接口（或其他后端接口）时，应该选择 `Promise.all` 还是 `Promise.allSettled`？它们的行为有什么区别？不同业务场景下如何选型？

---

## 💡 解题思路

1. 简要回顾 `all` 和 `allSettled` 的语义
2. 讨论「强一致」场景 vs 「尽可能多返回」场景
3. 结合多模型/多接口请求的典型需求说明选型
4. 提醒错误处理、超时控制等额外考量

---

## ✅ 面试回答（标准版）

1. **行为对比**  
   - `Promise.all(promises)`：  
     - **要求全部成功**：只要有一个 Promise 变为 `rejected`，整体就立刻 `reject`  
     - 成功时返回一个按输入顺序排列的结果数组  
     - 适合「全有或全无」的场景  
   - `Promise.allSettled(promises)`：  
     - **等待全部结束**：无论成功或失败都会等到所有 Promise 都 settle  
     - 永远不会 `reject`，而是返回一个数组，元素为 `{ status: 'fulfilled', value }` 或 `{ status: 'rejected', reason }`  
     - 适合「需要统计每个任务最终状态」的场景  

2. **多模型/多接口调用场景分析**  
   - **场景一：所有接口都是「必需」的强依赖**  
     - 例如：渲染页面前必须同时拿到「用户信息 + 权限配置 + 必要配置项」才能继续  
     - 任一接口失败都意味着当前业务无法继续，这时应使用 `Promise.all`：  

     ```js
     const [user, perms, config] = await Promise.all([
       fetchUser(),
       fetchPermissions(),
       fetchConfig(),
     ]);
     ```

     - 一旦有一个失败，统一走错误处理流程（如展示错误页、重试等）  

   - **场景二：部分接口是「可选」或「增强性」的数据**  
     - 例如：多个模型接口并发推理，能多返回几个结果最好，但失败几个也无妨；或主数据成功即可加载页面，其他是侧边推荐信息  
     - 此时更合适用 `Promise.allSettled`：  

     ```js
     const results = await Promise.allSettled([
       callModelA(),
       callModelB(),
       callModelC(),
     ]);

     const fulfilled = results.filter(r => r.status === 'fulfilled');
     // 根据成功的模型结果继续逻辑，失败的可以记录日志或降级
     ```

     - 即使有部分模型失败，也不会影响整体流程，可以基于成功的子集做最佳努力  

3. **扩展：错误处理与超时控制**  
   - 无论使用 `all` 还是 `allSettled`，实际项目中通常还需要：  
     - 设置单接口超时（例如 Promise.race 包装一个 timeout）  
     - 对失败结果做分级处理：重试、降级、告警等  
   - 对于「重要 + 可重试」的接口，可以在失败分支中增加重试逻辑后再决定整体行为  

4. **选型总结**  
   - **当多个请求「缺一不可」时，用 `Promise.all`**，失败视为整体失败  
   - **当多个请求「能多则好，部分失败也能继续」时，用 `Promise.allSettled`**，然后对每个结果分别处理  
   - 对于模型集合调用、推荐系统、多数据源聚合等场景，`allSettled` 通常更灵活、更鲁棒  

一句话：根据业务对「局部失败」的容忍度来决定，如果某个接口失败就要整体失败，选 `Promise.all`；如果希望拿到所有成功和失败的信息再做决策，选 `Promise.allSettled`。
