## 🧩 题目

如何实现 Promise 缓存（`cachePromise`）？它可以解决哪些问题？典型的应用场景和潜在缺点是什么？

---

## 💡 解题思路

1. 定义：同一入参的异步调用只发起一次请求，后续复用同一个 Promise
2. 核心实现：以参数为 key 的 Map，缓存 Promise；在请求完成后可选择保留或失效
3. 应用场景：接口幂等请求、配置拉取、去重并发、避免抖动等
4. 潜在问题：缓存过期策略、错误缓存、内存占用

---

## ✅ 面试回答（标准版）

1. **什么是 Promise 缓存 / `cachePromise`**  
   - 指的是：**对于相同参数的异步操作，在一段时间内只真正发起一次，后续调用直接复用第一次返回的 Promise**  
   - 典型用途是：避免同一时刻或短时间内对同一资源发起多次重复请求，既减少后端压力，也避免前端状态抖动  

2. **简化实现思路**  

```js
function createPromiseCache(fn) {
  const cache = new Map(); // key -> Promise

  return function cachedFn(key) {
    if (cache.has(key)) {
      return cache.get(key);
    }

    const p = Promise.resolve().then(() => fn(key));
    cache.set(key, p);

    // 可选：如果不希望失败结果被长期缓存，可以在失败时清理
    p.catch(() => cache.delete(key));

    return p;
  };
}
```

   - 使用方式示例：

   ```js
   const fetchUserCached = createPromiseCache(id =>
     fetch(`/api/user/${id}`).then(res => res.json())
   );

   // 下面多次并发调用只会触发一次真实请求
   const p1 = fetchUserCached(1);
   const p2 = fetchUserCached(1);
   ```

3. **典型应用场景**  
   - **去重并发请求**：页面多个组件同时需要同一份数据（如用户信息、配置项），可避免重复打接口  
   - **幂等/只读接口缓存**：一些配置、元数据更新频率较低，可以通过 Promise 缓存 + 失效策略减少请求次数  
   - **输入抖动下的稳定请求**：用户快速点击或频繁触发同一操作时，利用缓存 Promise 合并为一次请求  

4. **潜在缺点与注意事项**  
   - **缓存失效策略**：
     - 如果一直不清理，Map 可能不断增大，导致内存占用问题  
     - 可以增加 TTL（过期时间）或在业务层按需 `delete` 缓存键  
   - **错误结果是否缓存**：
     - 若失败的 Promise 也被缓存，相同参数的后续调用会立即得到失败结果，而不会重新尝试  
     - 通常在 `catch` 时把失败的 key 从缓存中删掉，以允许后续重试  
   - **参数作为 key 的序列化问题**：
     - 示例中直接用简单的 `key`，但在实际业务里参数可能是对象，需要可预测的序列化（如 JSON.stringify + 排序）  
   - **与浏览器缓存/HTTP 缓存的边界**：
     - Promise 缓存只是前端运行时级别的，刷新页面后会丢失，如果需要跨会话缓存需要结合 localStorage / IndexedDB 等  

5. **总结**  
   - Promise 缓存的核心思想是「**复用同一个进行中的 Promise**」，避免同一异步任务被重复触发  
   - 合理设计 key、失效策略和错误处理，可以在复杂异步场景下提升性能和用户体验，但也要谨慎防止缓存不一致和内存泄漏
