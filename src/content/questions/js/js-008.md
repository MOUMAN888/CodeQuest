## 🧩 题目

`Object.is()` 与 `==`、`===` 的区别是什么？在实际项目中应如何选择正确的「相等性判断」方式？

---

## 💡 解题思路

1. 解释 `==`（宽松相等）：会进行类型转换
2. 解释 `===`（严格相等）：不做隐式类型转换，但对 `NaN` 和 `+0/-0` 有特殊情况
3. 解释 `Object.is`：与 `===` 类似，但修复 `NaN`、`+0/-0` 的两个边界问题
4. 总结工程实践中推荐的使用规则

---

## ✅ 面试回答（标准版）

1. **`==`（宽松相等）**  
   - 会在比较前**尝试进行类型转换**，规则很复杂，容易出 Bug：  
     - `'1' == 1 // true`（字符串转数字）  
     - `true == 1 // true`、`false == 0 // true`  
     - `null == undefined // true`  
   - 由于隐式转换规则复杂且不易记忆，实际项目中通常不推荐使用，除非特别明确需要 `null` / `undefined` 合并判断（例如 `x == null` 同时判断两者）  

2. **`===`（严格相等）**  
   - 不做任何隐式类型转换，要求**类型相同且值相同**才返回 `true`：  
     - `'1' === 1 // false`  
     - `0 === false // false`  
   - 但在两个边界案例上存在「不直观」的行为：
     - `NaN === NaN // false`（NaN 与任何值都不相等，包括自身）  
     - `+0 === -0 // true`（区分不了正零和负零）  

3. **`Object.is` 的行为**  
   - `Object.is(value1, value2)` 提供了一个更“语义化”的相等判断规则：  
     - 除了两个特殊情况外，它与 `===` 的行为**完全一致**  
   - 修复的两个边界情况：
     - `Object.is(NaN, NaN) // true`（认为 NaN 等于 NaN）  
     - `Object.is(+0, -0) // false`（可以区分正零和负零）  
   - 对于大部分普通值，`Object.is(a, b)` 与 `a === b` 结果相同  

4. **如何选择？工程实践建议**  
   - **默认首选 `===`**：避免隐式类型转换导致的不可预期行为  
   - 需要判断 `NaN` 时：
     - 使用 `Number.isNaN(value)` 或 `Object.is(value, NaN)`  
   - 需要区分 `+0` 和 `-0`（通常在数值底层库中才会考虑）：
     - 使用 `Object.is(+0, -0)`  
   - 极少数情况下需要「同时把 `null` 和 `undefined` 当成空值」时，可以考虑使用 `x == null` 这种有意识的宽松比较  

总结：  
**`==` 容易出坑，除非非常明确语义，否则尽量避免；`===` 是日常开发的默认选择；`Object.is` 在需要处理 `NaN` 和区分 `+0/-0` 这类边界情况时非常有用。**
