## 🧩 题目

为什么 JavaScript 中 `0.1 + 0.2 !== 0.3`？底层原因是什么？在业务代码中如何更精准地处理浮点数运算精度问题？

---

## 💡 解题思路

1. JS `number` 类型基于 IEEE 754 双精度浮点数
2. 十进制小数在二进制中无法被有限位精确表示，存在舍入误差
3. 说明 `0.1 + 0.2` 实际结果接近 `0.30000000000000004`
4. 常见解决思路：放大为整数计算、使用 `toFixed`/`toPrecision`、使用专门的大数/高精度库

---

## ✅ 面试回答（标准版）

1. **根本原因：IEEE 754 双精度浮点数的表示限制**  
   - JavaScript 中的 `number` 类型（不含 `bigint`）统一采用 **IEEE 754 双精度浮点数**（64 位）表示  
   - 在这种表示方式下，并不是所有十进制小数都能用有限的二进制小数精确表达  
   - `0.1`、`0.2` 这类十进制小数在转换为二进制时会变成**无限循环小数**，只能截断/舍入到有限位  
   - 这就会引入非常细微的误差，导致：

   ```js
   0.1 + 0.2 === 0.30000000000000004; // 实际结果接近这个值
   ```

2. **JS 中常见的精度问题表现**  
   - 比较小数时，`===` 会因为这些微小误差而失败  
   - 在金额、统计等场景中，多次运算后误差会被放大  
   - 使用 `toFixed` 时可能看到看似「奇怪」的四舍五入结果  

3. **常见解决方案**  
   - **方案一：放大为整数再计算**  
     - 将所有参与运算的小数统一放大为整数（按精度乘以 10、100、1000 等），计算完成后再缩小：

     ```js
     function add(a, b) {
       const m = 10 ** 2; // 假设保留两位小数
       return (Math.round(a * m) + Math.round(b * m)) / m;
     }

     add(0.1, 0.2); // 0.3
     ```

     - 关键点在于统一放大倍数，避免在放大过程中引入额外误差  
   - **方案二：使用误差容忍度做比较**  
     - 比较两个浮点数是否「相等」时，引入一个很小的 `EPSILON`：

     ```js
     function nearlyEqual(a, b, epsilon = Number.EPSILON) {
       return Math.abs(a - b) < epsilon;
     }
     ```

     - 适合判断结果是否足够接近，而非严格等于  
   - **方案三：使用高精度/大数库**  
     - 对于金额、金融等对精度极为敏感的场景，建议使用如 `decimal.js`、`big.js` 等高精度运算库  
     - 通过内部字符串或大整数实现十进制精确计算，牺牲一定性能换取精度和可靠性  

4. **实践建议**  
   - 明确：浮点数在大部分语言中都存在类似问题，不是 JS 独有  
   - 在金额等敏感场景下，**不要直接依赖原生小数运算 + 严格相等比较**，应统一采用整数化/高精度库方案  

理解这一点的关键是：**JS 的小数只是近似值**，一旦涉及到比较和展示就要有「误差处理」的意识。
