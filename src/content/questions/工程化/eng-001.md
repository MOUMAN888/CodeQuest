## 🧩 题目

在团队开发中，Code Review 的频率应该如何设定比较合适？常见的 Git 分支模型（如 Git Flow、Trunk-Based 等）有哪些特点？

---

## 💡 解题思路

1. 回答 Code Review 频率：小步提交、尽量日常化，每个功能/缺陷均需 PR+Review
2. 列举常见分支模型：Git Flow、GitHub Flow、GitLab Flow、Trunk-Based
3. 简要分析各自适用场景与优缺点

---

## ✅ 面试回答（标准版）

1. **Code Review 频率与粒度建议**  
   - **频率**：  
     - 最好做到「**日常化、持续化**」的 Code Review，而不是大版本上线前一次性集中 Review。  
     - 建议每个功能/缺陷修复都通过 Pull Request（Merge Request）提交，并至少由 1~2 名同事 Review 后合并。  
   - **粒度**：  
     - 更推荐**小而频繁的 PR**（例如几十到一两百行代码），而不是一次性上千行的大型变更。  
     - 小 PR 更容易被认真审查，也更方便回滚和定位问题。  
   - 实际执行时，可根据团队规模和节奏设定规范：  
     - 如「所有业务代码变更至少需要 1 个 Review + CI 通过」  
     - 核心模块或架构变更需要更多 Reviewer 或资深同事参与。  

2. **常见 Git 分支模型与特点**  
   - **Git Flow**：  
     - 经典模型，分支包括：`master`（生产）、`develop`（开发主干）、`feature/*`、`release/*`、`hotfix/*` 等。  
     - 特点：流程清晰，适合有明显版本节奏的项目（如发布大版本、维护多个线上版本）。  
     - 缺点：分支较多、操作相对复杂，不太适合高频、小步快迭代的团队。  

   - **GitHub Flow**（简化模型）：  
     - 通常只有一个长期存在的主分支 `main/master`，每个功能从主分支拉 `feature` 分支开发，完成后通过 PR 合并回主分支并部署。  
     - 特点：简单直接，适合持续交付的 Web 项目。  
     - 要求主分支始终保持可部署状态。  

   - **GitLab Flow**：  
     - 在 GitHub Flow 的基础上引入环境分支（如 `production`、`staging`）和「上游优先」理念。  
     - 针对不同部署环境（预发/生产）建立不同分支，合并策略更贴近 CI/CD 流程。  

   - **Trunk-Based Development（主干开发）**：  
     - 所有人围绕一个主干分支（`main`）短生命周期地创建小分支开发，尽快合并回主干。  
     - 大部分特性在主干上通过「特性开关（Feature Toggle）」控制发布，而不是长期挂起的大分支。  
     - 特点：非常契合持续集成/持续交付实践，减少大规模分支合并冲突。  
     - 要求自动化测试和 CI 体系较完善，且团队有较强的工程纪律。  

3. **组合实践建议**  
   - 中小型前端团队常用组合：  
     - 以 **GitHub Flow / Trunk-Based** 为基础：`main` 作为唯一长期分支，每个需求开一个 `feature/*` 分支，通过 PR + Code Review 合并，并使用 CI 校验。  
     - 若有复杂发布节奏，可适当引入 `release/*` 或环境分支。  
   - 不论采用哪种模型，都建议明确：  
     - 分支命名规范、合并策略（Squash / Rebase）、Review 流程、CI 必经步骤等。  

一句话：Code Review 应该成为日常持续活动，以小而频繁的 PR 配合合适的分支模型（如简化的 GitHub Flow 或主干开发），在保证质量的同时不拖慢交付节奏。
