## 🧩 题目

在虚拟 DOM 的 Diff 算法中，`key` 的作用是什么？为什么不推荐使用数组的 `index` 作为 `key`？在实际项目中应该如何选择 `key`？

---

## 💡 解题思路

1. 解释 key 的作用：标识节点身份，帮助 Diff 做最小化移动/复用
2. 使用 index 作为 key 的问题：插入/删除/排序会导致错误复用，产生状态错乱和性能问题
3. 举常见例子：带输入框/动画的列表使用 index 作为 key 的 bug
4. 总结选择 key 的原则：稳定、唯一、与数据强相关，尽量避免 index

---

## ✅ 面试回答（标准版）

1. **Diff 算法中 `key` 的作用**  
   - 在 React 的虚拟 DOM Diff 过程中，`key` 用来**唯一标识同一层级的兄弟节点**，帮助 React 判断：  
     - 哪些节点可以复用（只是 props 变化）  
     - 哪些节点需要插入、删除或移动  
   - 有了稳定的 `key`，React 可以在列表「增删改」中做到更精确的最小化更新，而不是简单按顺序一一对比  

2. **为什么不推荐使用数组 `index` 作为 `key`？**  
   - 当列表**不会发生重排、插入、删除**时，用 `index` 问题不大，但一旦有这些操作，就容易出问题：  
     - React 会认为「位置没变、`key` 没变」的节点是「同一个节点」，从而复用其内部状态和 DOM  
     - 如果你在中间插入或删除一项，后面的所有项的 index 都会改变，导致 React 把每一项都当成「别人」来复用  
   - 典型现象：  
     - 在一个带输入框的列表中，以 index 作为 key，当你在中间插入/删除一行时，后面行的输入内容可能会「串行」，因为 React 复用了错误的 DOM 节点和内部 state  
     - 在带动画/过渡的列表中，动画会错乱或出现在错误的项上  

3. **示例：index 作为 key 的潜在 bug**  

```jsx
{list.map((item, index) => (
  <input
    key={index}
    defaultValue={item.text}
  />
))}
```

   - 当你在列表头部插入一个新元素时，原来的第 0 行变成第 1 行，但它们的 key 都是 0/1/2 递推变化  
   - React 会认为「原来的第 1 行节点现在是第 2 行」，于是复用 DOM 和 state，导致用户在某一行输入的内容跑到了另一行  

4. **正确选择 `key` 的原则**  
   - **稳定且唯一**：`key` 应该是能在整个列表生命周期中稳定标识同一条数据的 id，比如数据库自增 id、唯一标识符等  
   - **与数据强相关而不是与渲染位置相关**：  
     - 好的 key：`item.id`、`item.uuid`  
     - 不好的 key：数组 index、随机数（每次 render 生成新 random 也会导致 diff 全部失败复用）  
   - 当无法提供稳定 id 时（如静态小列表），可以在确认「不会插入/删除/重排」的前提下使用 index，但要非常明确这一前提  

5. **总结**  
   - `key` 的作用是帮助 React 正确、高效地识别节点身份，完成最小化的 DOM 更新  
   - 使用 index 作为 key 容易在「可变列表」中造成节点错位、状态串行和性能退化，实际开发中应尽量使用与业务数据强绑定、稳定不变的唯一标识作为 key。
