## 🧩 题目

React 的服务端渲染（SSR）原理是什么？与纯客户端渲染（CSR）相比，有哪些优势和缺点？大致的渲染流程是怎样的？

---

## 💡 解题思路

1. 定义 SSR：在服务端把 React 组件渲染为 HTML 字符串返回浏览器
2. SSR 与 CSR 渲染流程对比：首屏 HTML、hydrate 过程
3. 优势：更快的首屏渲染、SEO 友好、统一渲染逻辑
4. 缺点：实现复杂度、服务端压力、缓存与状态管理挑战

---

## ✅ 面试回答（标准版）

1. **React SSR 的基本原理**  
   - 在纯客户端渲染（CSR）模式下，浏览器拿到的是一个几乎空的 `index.html` 和一个 JS 包，必须先下载/解析/执行 JS，才能在客户端构建虚拟 DOM 并生成真实 DOM  
   - 在服务端渲染（SSR）模式下，**由服务器提前运行 React 组件树，生成完整的 HTML 字符串返回给浏览器**，浏览器拿到的 HTML 就已经包含了首屏的 DOM 结构和内容  
   - 前端 JS 包下载并执行后，会在这份已有的 DOM 之上进行 **hydrate（注水/激活）**，把事件绑定、状态等挂载上去，使之变成可交互的 React 应用  

2. **SSR 的渲染流程（简化版）**  
   - 请求到达服务端时：  
     1）根据 URL 和路由匹配需要渲染的 React 组件  
     2）在服务端获取首屏所需数据（调用 API 等）  
     3）使用 `ReactDOMServer.renderToString`（或 `renderToPipeableStream` 等新 API）将组件树渲染为 HTML 字符串  
     4）把渲染好的 HTML 和预注入的首屏数据（如 `window.__INITIAL_STATE__`）一起返回给浏览器  
   - 浏览器端：  
     1）用户立刻看到完整的 HTML（首屏更快）  
     2）下载并执行前端 JS 代码  
     3）React 在已有 DOM 上执行 `hydrateRoot` / `hydrate`，复用 DOM 节点并挂载事件监听 / 状态，使页面进入可交互状态  

3. **与 CSR 相比的优势**  
   - **首屏渲染更快**：尤其在网络/设备较差时，用户无需等待 JS 执行完成就能看到内容  
   - **SEO 友好**：搜索引擎直接获取到完整 HTML，有利于内容索引（尽管现代搜索引擎也支持执行 JS，但 SSR 更稳定可控）  
   - **统一渲染逻辑**：前后端共享同一套 React 组件和路由逻辑，便于维护  

4. **SSR 的缺点与挑战**  
   - **实现复杂度更高**：  
     - 需要处理「同构」（isomorphic）代码：既能在服务端运行，又能在浏览器运行  
     - 需要避免在服务端直接访问浏览器 API（如 `window`、`document`）  
   - **服务器压力更大**：  
     - 每次请求都要在服务端执行 React 渲染和数据请求，相比纯静态托管增加了 CPU 和 IO 开销  
     - 需要更精细的缓存策略（如页面缓存、数据缓存）  
   - **调试和部署更复杂**：  
     - 涉及 Node 端和浏览器端两套运行环境  
     - 错误堆栈、性能监控等也更复杂  

5. **适用场景与权衡**  
   - SSR 适合：内容型网站（博客、文档、营销页）、首屏性能要求高的 SPA、对 SEO 有明显需求的应用  
   - 完全 CSR 适合：SEO 不敏感、更多是登录态应用或内部后台系统  
   - 还可以结合预渲染（Pre-rendering）、SSG（静态站点生成，如 Next.js 的静态导出），在不同页面上做不同渲染策略的组合  

一句话：React SSR 的核心是「在服务器先把组件渲染成 HTML，再在客户端 hydrate」，带来更好的首屏体验和 SEO，但也引入了实现复杂度和服务器资源消耗，需要结合业务场景权衡使用。
