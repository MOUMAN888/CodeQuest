## 🧩 题目

React 的合成事件（SyntheticEvent）有什么作用？它与原生 DOM 事件有哪些区别？底层大致是如何实现的？

---

## 💡 解题思路

1. 合成事件的目的：跨浏览器兼容、统一事件对象、节省内存、方便事件委托
2. 与原生事件的区别：事件绑定位置、命名方式、事件池（老版本）、阻止默认行为方式
3. 底层实现：事件委托到根节点（`document` / `root`），统一分发回调
4. 说明新版 React 事件系统的一些变化（如 17+ 绑定到根容器）

---

## ✅ 面试回答（标准版）

1. **合成事件的作用**  
   - React 并不是直接把你写在 JSX 上的事件处理函数（如 `onClick`）绑定到真实 DOM 元素上，而是通过一层 **合成事件系统（SyntheticEvent）** 来管理：  
     - 封装和**标准化**不同浏览器间的事件对象和行为（如 `event.target`、`event.preventDefault()` 等）  
     - 通过 **事件委托** 统一在顶层节点监听，减少实际绑定在 DOM 上的事件处理器数量，节省内存  
     - 便于在内部做事件优先级控制、批量更新、兼容性处理等  

2. **与原生 DOM 事件的主要区别**  
   - **绑定位置不同**：
     - 原生事件：你使用 `element.addEventListener` 时，是直接绑在具体 DOM 节点上  
     - React 合成事件：大多数事件会被**统一绑定在根容器（如 `document` 或 React 17+ 的根节点）上**，通过事件冒泡捕获，再按内部注册表找到对应组件的回调  
   - **事件对象不同**：  
     - React 事件处理函数收到的是 `SyntheticEvent` 实例，而不是浏览器原生事件对象（但内部持有 `nativeEvent` 引用）  
     - 它对一些浏览器差异做了封装，并提供一致的 API  
   - **事件命名方式**：
     - JSX 中用的是**小驼峰**写法：`onClick`、`onChange`，而非原生的 `onclick` 属性  
   - **事件池机制（老版本）**：  
     - 在 React 17 以前，`SyntheticEvent` 对象使用事件池复用，事件回调调用结束后属性会被清空，需要及时读取或调用 `event.persist()`  
     - 新版本中这一行为已被移除，事件对象不再被池化，减少了常见坑  

3. **底层实现原理（简化版）**  
   - 在初始化阶段，React 会在根容器节点上注册一系列原生事件监听器（如 `click`、`change` 等）  
   - 当这些原生事件触发时：  
     1）浏览器事件冒泡到根容器，被 React 的统一监听器捕获  
     2）React 根据事件目标节点和内部维护的 Fiber 树，向上查找所有注册了对应合成事件处理函数的组件  
     3）创建一个 `SyntheticEvent` 对象，填充统一的属性与方法  
     4）按从内到外的顺序依次调用对应的 React 事件回调（模拟捕获/冒泡阶段）  
     5）根据 React 的批量更新策略，在合适的时机一起刷新 UI  

4. **总结**  
   - 合成事件本质上是 React 在原生事件之上的一层抽象，用来**做跨浏览器兼容 + 事件委托 + 批量更新控制**  
   - 对开发者而言，大部分情况下可以把它当「稍作封装的 DOM 事件」来用；只有在与原生事件系统交互（如 `addEventListener`、第三方库）时，才需要特别注意二者的差异和交互方式。
