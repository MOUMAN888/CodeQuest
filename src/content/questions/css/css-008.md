## 🧩 题目

说说你对 CSS 工程化的理解？在中大型项目中，通常会从哪些方面对 CSS 做工程化管理？

---

## 💡 解题思路

1. 痛点：全局污染、命名冲突、样式难以维护与复用
2. 组织方式：命名规范（BEM）、模块化（CSS Modules）、组件化（CSS-in-JS、单文件组件）
3. 工具链：预处理/后处理（Sass、PostCSS）、构建工具（打包、压缩、自动添加前缀）
4. 质量保障：规范检查（stylelint）、设计系统/设计令牌（Design Tokens）

---

## ✅ 面试回答（标准版）

所谓 CSS 工程化，本质上是把「写样式」这件事当成工程来做，通过**规范、工具和架构**来解决中大型项目中 CSS 易污染、难维护、难复用的问题。可以从几个维度来理解：

1. **代码组织与命名规范**  
   - 使用合理的命名规范（如 **BEM：Block / Element / Modifier**），让类名本身就能表达结构和状态，降低命名冲突概率：  
     - `btn`（Block）  
     - `btn__icon`（Element）  
     - `btn--primary`（Modifier）  
   - 按业务模块、组件模块拆分样式文件，而不是一个巨大的全局 CSS  

2. **CSS 模块化与组件化**  
   - **CSS Modules**：通过构建将类名局部哈希化，实现样式作用域隔离，如 `styles.btn`  
   - **单文件组件（SFC）**：在 Vue / Svelte 等框架中，`<style scoped>` 让样式只作用于当前组件  
   - **CSS-in-JS**：如 styled-components、Emotion，把样式与组件逻辑组合在一起，通过 JS 动态生成样式，便于主题切换、按需加载等  

3. **预处理与后处理**  
   - 预处理器：Sass / Less / Stylus 提供变量、嵌套、Mixin、函数等能力，提高复用性和可维护性  
   - 后处理器：PostCSS + autoprefixer 自动补全厂商前缀，配合 `postcss-preset-env` 使用新特性  
   - 使用构建工具（Webpack、Vite、Rollup 等）进行打包拆分、代码压缩、提取公共样式等  

4. **规范与质量保障**  
   - 引入 **stylelint** 等工具进行静态检查，约束命名风格、禁止 !important 滥用、禁止深度嵌套  
   - 结合设计系统/设计规范：抽象颜色、字体、间距为 **Design Tokens**，统一通过变量或 CSS 变量（`var(--color-primary)`）管理  
   - 在组件库层面复用：将常用布局和 UI 模块沉淀为可复用组件，而不是在每个页面重复写 CSS  

5. **性能与交付优化**  
   - 按需加载样式：避免首屏加载过多无关 CSS  
   - 移除未使用 CSS（tree-shaking / PurgeCSS 等）  
   - 利用 `critical CSS` 技术内联关键样式，提升首屏渲染速度  

总结来说，CSS 工程化并不局限于某个具体技术，而是一整套围绕「**可维护、可扩展、可协作**」目标的实践集合：从命名规范到模块化方案，从预处理/构建工具到检查工具，再到设计系统和性能优化，共同支撑复杂项目中的样式管理。
