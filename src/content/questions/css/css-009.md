## 🧩 题目

`z-index` 属性在什么情况下会「看起来失效」？请结合层叠上下文（Stacking Context）进行解释。

---

## 💡 解题思路

1. 先说明 `z-index` 只对「定位元素」且在同一层叠上下文中比较有效
2. 介绍什么是层叠上下文：谁会创建新的 Stacking Context
3. 典型「失效」场景：不在同一上下文、父元素被覆盖、`z-index` 为 `auto` 等
4. 总结排查思路：先看定位、再看层叠上下文，再对比层级

---

## ✅ 面试回答（标准版）

很多人觉得 `z-index`「没生效」，其实大多数情况不是失效，而是**被层叠上下文规则限制了比较范围**。理解 `z-index`，一定要先理解「层叠上下文」（Stacking Context）。

1. **`z-index` 生效的前提**  
   - 一般只对**定位元素**有效（`position: relative/absolute/fixed/sticky`），非定位元素设置 `z-index` 通常不起作用  
   - 元素的 `z-index` 只能在**同一个层叠上下文**中进行比较，不同上下文之间先比较「整个上下文」的层级  

2. **什么是层叠上下文（Stacking Context）**  
   可以理解为浏览器在绘制时的一层「小世界」：  
   - 每个层叠上下文内部有自己的 z 轴排序规则  
   - 不同层叠上下文之间，整体按父级的层级参与比较  

   典型的创建层叠上下文的方式包括：  
   - 根元素（`<html>`）天然是一个层叠上下文  
   - `position` 非 `static` 且 `z-index` 非 `auto`  
   - `opacity < 1`、`filter`、`transform`、`will-change` 等某些 CSS 属性  
   - `isolation: isolate`、`mix-blend-mode` 非 `normal` 等  

3. **常见「z-index 失效」场景**  
   - **不在同一个层叠上下文**  
     - A 元素设置 `z-index: 9999`，但它的父元素所在的层叠上下文整体层级低于另一个兄弟元素所在的层叠上下文 → 再高也压不过去  
   - **父元素被遮挡**  
     - 子元素的 z 轴层级不会突破父元素所在的层叠上下文，父元素在别的元素下面时，子元素也没法「飞到外面去」盖过它们  
   - **元素本身没有建立可比较的层级**  
     - 如元素 `position: static`，设置 `z-index` 实际无效  
     - 或者 `z-index: auto`，仍然按默认的层叠顺序参与比较  

4. **排查与解决思路**  
   - **第一步**：检查是否设置了 `position`，避免在 `static` 上调 `z-index`  
   - **第二步**：用开发者工具查看元素所属的层叠上下文（某些浏览器有可视化支持），看是否被父级的 `transform`、`opacity` 等创建了新的上下文  
   - **第三步**：必要时将需要「压在最上面」的元素移动到更外层，或让相关元素处于同一个层叠上下文下再比较 `z-index`  

总结：`z-index` 并不是一个「全局绝对高度」，它只能在自己的层叠上下文里生效。如果理解了谁会创建新的层叠上下文，以及父子/兄弟上下文之间的比较顺序，很多看似「失效」的问题就能解释清楚。
